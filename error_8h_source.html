23 // Generally, we consider three types of errors: Machine errors, Programming errors, and User errors
24 //
25 // Machine errors occur when the hardware is incapable of handling some operation. The most common
26 // example of this is out of memory errors. Our typical response to these is to panic and exit, which is
27 // triggered by allocator->out_of_memory().
28 //
29 // Programming errors should notify the developer that something is incorrect, and should probably
30 // interfere with compilation (or at least give a warning) until the issue is resolved. We handle these
31 // with log messages, as in log__throw_fatal_error().
32 //
33 // User errors are only detectable at runtime, and include issues such as improper input - file not found,
34 // or invalid formatting, for example. The proper way to handle these is to notify the user, and gracefully
35 // exit the program.
36 //
37 // Handling user errors is the purpose of this structure.
38 //
39 // The type field is the broad category of error which was encountered. This is generally the name of the
40 // class which defines and throws the error.
41 //
42 // The code field is an integral identifier for the specific error encountered
43 //
44 // The message field allows the developer to provide a detailed message describing the error and relevant
45 // conditions of the error when encountered
46 //
47 // By nature, user errors bubble up to the parent scope. This means that the top-level caller must free
48 // allocated memory once it's no longer needed. There are a few approaches to handling this.
49 //
50 // One approach is to stack-allocate all errors and their data. In this case the parent scope does not have
51 // to free any memory, because it was not dynamically allocated. This is not ideal, however, because it either
52 // limits the amount of contextual information that can be included in the message field, or it uses the
53 // message field memory inefficiently.
54 //
55 // Another approach is to heap-allocate all errors that are passed up to a parent caller. In this way, the parent
56 // can free the memory allocated for the error after it has dealt with the error condition. This is the approach
57 // we use here. We store a reference to the allocator in the error itself, that way the parent is able to free the
58 // memory even if a different allocator was used to allocate the error.
59 //
60 // In addition, we can still flag the error's memory as stack-allocated by setting the allocator field to NULL.
